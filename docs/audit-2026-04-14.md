# Hydra Audit — 2026-04-14

Scope: audit + fix + improve the P1 shipment against the referenced
papers (EnIGMA 2409.16165, Cybench 2408.08926, NYU CTF Bench
2406.05590, Palisade 2412.02776, CAI 2504.06017).

State before audit: **64 tests passing**.
State after audit:  **73 tests passing**.

---

## Critical issues found (must fix)

### C1. Auth required API key only — no Claude Max subscription path
`docker_worker.run_worker` took `api_key: str` as required. Anyone with
a Claude Code subscription but no raw API key could not use Hydra.
**Status:** FIXED. Worker now accepts `credentials_dir: Path | None`
(bind-mounts host `~/.claude` at `/root/.claude:ro`) OR `api_key`.
Subscription auth is preferred; API key is fallback. CLI auto-detects.
Test matrix covers both paths (`tests/unit/test_docker_worker.py`,
`tests/unit/test_cli.py`).

### C2. Specialist prompts referenced files that don't exist
`rev-specialist.md` pointed at `exploits/rev/patch_no_jmp.py` and
`exploits/rev/angr_find_input.py`. Neither file exists (P1 templates
placed under `exploits/pwn/`). The agent would read a broken reference
on its first solve attempt.
**Status:** FIXED. References corrected to a concrete r2 command and to
`exploits/pwn/angr_find_input.py` (explicitly noted as reusable for rev).
Validated: zero dangling references across all 6 specialists.

### C3. Orchestrator collapsed `error` into `failed`
Spec defined distinct statuses (`solved | failed | timeout | error`);
P1 collapsed non-zero-exit into `failed` to satisfy an inconsistent
test. Debuggers couldn't distinguish "worker crashed" from "worker ran
and found no flag".
**Status:** FIXED. `hydra/orchestrator.py` restored to 4-status logic.
`test_error_status_on_nonzero_exit` and `test_failed_status_on_clean_exit_no_flag`
guard both branches. Integration test `test_mixed_batch` asserts the
exit_code=1 case is `error`.

## High concerns (should fix)

### H1. No persistent-session support (EnIGMA / NYU)
EnIGMA (2409.16165) and NYU CTF Bench (2406.05590) document that pwn
challenges requiring multi-turn stateful interaction (gdb, long-lived
nc) need persistent sessions. A fresh `nc` per Bash call drops state.
**Status:** FIXED.
- Added `tmux`, `socat`, `netcat-openbsd` to `docker/apt-packages.txt`.
- Wrote `.claude/skills/pwn/tmux-session.md` with the concrete
  `tmux new-session -d` / `send-keys` / `capture-pane` recipe.
- Wrote `.claude/skills/meta/iat-pattern.md` generalizing the pattern
  across gdb, r2 interactive, msfconsole, sqlmap shell, Python/sage REPL.
- pwn-specialist now references the skill.

### H2. No subtask-decomposition playbook (Cybench)
Cybench (2408.08926) showed decomposition roughly doubles hard-challenge
solve rate. Hydra's triage agent never prompted the specialists to
write a plan for multi-stage challenges.
**Status:** FIXED.
- Wrote `.claude/skills/meta/subtask-decomposition.md` with a
  `./work/plan.md` template.
- CLAUDE.md now points specialists at the skill when the task feels
  multi-stage.

### H3. No "shell-first" principle (Palisade)
Palisade (2412.02776) surprising result: plain-ReAct + shell is
competitive with heavy tool stacks. Our specialists led with heavy
tooling (pwntools, sage, ghidra), which encouraged premature complexity.
**Status:** FIXED. All 6 specialists now open with a
"Top principle: shell-first, <heavy-tool>-last" section showing concrete
one-liners (strings, curl, exiftool, ltrace, base64) to try *before*
reaching for the big tools.

### H4. No cross-run memory (CAI)
CAI (2504.06017) emphasizes persistent learning across runs. Hydra had
no mechanism for specialists to record gotchas or reusable knowledge.
**Status:** FIXED.
- Added `notes/lessons-learned.md` scaffold, committed.
- CLAUDE.md instructs agents to append a one-line lesson after
  nontrivial solves.
- Format: `YYYY-MM-DD  [category]  <one-line lesson>`.

### H5. Lint noise on pwntools-convention exploits
`from pwn import *` is the universally-accepted pwntools convention in
CTFs but triggers F403/F405 ruff warnings on every reference. Without
config, every lint run produces 23 false positives.
**Status:** FIXED. `pyproject.toml` adds per-file ignores for
`exploits/**/*.py`. `ruff check .` now returns clean.

### H6. `test_filename_collision_suffixes` used E702 (semicolon)
Cosmetic but ruff-flagged.
**Status:** FIXED.

## Medium concerns (consider fixing)

### M1. No Docker build verification yet
Docker is not installed on the dev host, so the Dockerfile has never
actually been built. Layer ordering, apt-pinning, and `claude -p`
handshake could all fail on first real build. The integration tests
use a fake-docker shim, which verifies orchestrator correctness but not
image correctness.
**Mitigation:** build the image as soon as Docker is installed. Expect
the first build to need 1-2 minor fixes (NodeSource URL drift is the
typical culprit).
**Status:** LISTED, NOT FIXED.

### M2. No real E2E tests with canned challenges
Plan P1 deliberately deferred the 5-canned-challenge E2E suite to a
later pass. We have no evidence the end-to-end system solves real
trivial CTFs.
**Status:** LISTED (known deferral, tracked in the P1 plan).

### M3. Bleeding-edge Python on host (3.14)
Dev host has Python 3.14.4; spec targets 3.12+. Works today but any
breaking-change Python release could bite. Pin a minimum in CI once
that exists.
**Status:** LISTED.

### M4. Dockerfile layer ordering is sub-optimal
`npm install -g @anthropic-ai/claude-code` is the top layer (correct —
most churn), but `RUN apt-get install ffuf gobuster ...` is split across
multiple RUN statements. Each RUN adds an image layer. Consolidating
apt installs would save ~100 MB and speed up builds.
**Status:** LISTED, NOT FIXED (micro-optimization).

### M5. `resolve_config` has a 12-field dataclass
`ResolvedConfig` bundles challenges, auth, paths, flags, limits into one
struct. Reasonable for v1 but will get unwieldy if more options land.
**Status:** LISTED, NOT FIXED.

### M6. No cost/token tracking in user-facing output
`results.jsonl` records per-challenge duration and exit code but not
token usage or cost. During a live comp, a user might want to know
"how much have I spent so far".
**Status:** LISTED, NOT FIXED (not requested; logged internally only).

### M7. No retry/backoff for transient `docker run` failures
If the Docker daemon is restarting, a worker spawn will fail once and
the whole challenge is marked `error`. A 1-retry-after-backoff would
raise availability.
**Status:** LISTED, NOT FIXED.

### M8. `flag_extractor` doesn't surface flag ambiguity
If stdout contains multiple plausible flags (e.g., an unrelated string
matching the generic regex earlier + a real flag later), the last-match
rule picks the right one but silently drops the rest. An ambiguity
warning in the log would help debug false-positive flag cases.
**Status:** LISTED, NOT FIXED.

### M9. `_looks_logged_in` is heuristic
Checks for `credentials.json`/`settings.json` presence in `~/.claude`.
A stale install without valid creds would pass this check and then fail
at `claude -p` inside the container. Better: try `claude --version`
on the host at startup to verify actual auth.
**Status:** LISTED, NOT FIXED.

### M10. No input-path validation before docker spawn
If a challenge JSON points at `/path/that/does/not/exist`, we create the
workdir and spawn the worker — the worker ultimately reports "no files",
but a pre-flight fail-fast would save ~a minute per affected challenge.
`workdir.build_workdir` logs a warning but proceeds; the agent may do
nothing useful without the file.
**Status:** LISTED, NOT FIXED (current behavior is forgiving by design,
but could add a strict mode).

### M11. 2 agents in `.claude/agents/` have subtle front-matter drift
pwn-specialist ~85 lines (incl. shell-first), misc-specialist ~56 lines.
Slight inconsistency in length / structure across the 6 files. Not
blocking, but a polish pass could align headers.
**Status:** LISTED.
